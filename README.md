# React Movie Gallery

Индивидуальный веб-проект (SPA) для поиска, фильтрации и каталогизации кинофильмов с интеграцией облачного бэкенда и LLM-ассистента для генерации персональных рекомендаций.

## Ссылка на проект
Живое демо: https://movie-gallery-rose.vercel.app

## Основной технологический стек
* **Фронтенд:** React 19 (Hooks, Context API, Lazy/Suspense), React Router 7, Vite 7.
* **Бэкенд (BaaS):** Supabase (PostgreSQL, Auth, Edge Functions).
* **Интеграции:** Kinopoisk Unofficial API, Groq API (LLM-инференс).
* **Тестирование:** Vitest, React Testing Library.
* **Линтинг:** ESLint.

## Ключевые архитектурные решения и функционал

### 1. Управление состоянием и бизнес-логика
* Вся работа с данными (каталог, списки пользователя, темы, авторизация) вынесена в специализированные контексты (React Context), что предотвращает избыточный проп-дриллинг.
* Интерфейсы ввода и сетевые запросы изолированы в кастомных хуках (`useFetch`, `useForm`, `useModal`, `useFilter`), обеспечивая переиспользуемость кода и разделение ответственности (Separation of Concerns).

### 2. Синхронизация данных и безопасность (BaaS)
* Реализована полноценная аутентификация через Supabase Auth (Email/Password, Google OAuth).
* Списки «Избранное» и «Просмотрено» хранятся в PostgreSQL на стороне Supabase и синхронизируются в реальном времени только для авторизованных сессий.
* Доступ к административным возможностям (добавление и редактирование кастомных фильмов) разграничен на основе ролевой модели (RBAC) через пользовательские метаданные.

### 3. Интеграция ИИ (Инженерия промптов)
* Внедрен интерактивный чат-ассистент на базе Groq API. Для экономии лимитов и защиты API-ключей запросы на фронтенде закрыты авторизационным барьером, а взаимодействие с LLM вынесено на уровень изолированных серверных функций. Вектор ответов жестко ограничен валидным JSON-форматом для последующего парсинга фронтендом.

### 4. Оптимизация производительности
* Для минимизации размера стартового бандла и ускорения Time to Interactive (TTI) применено разделение кода (Code Splitting) через `React.lazy` и компоненты `Suspense` для всех страниц и ресурсоемких модальных окон.

## Структура репозитория

```text

├── api/          # Модули инициализации клиентов (Supabase, Kinopoisk, Groq)

```

```bash

```

## Доступные скрипты

* `npm run dev` — запуск локального окружения разработки.
* `npm run build` — компиляция и оптимизация проекта в production-бандл.
* `npm run preview` — локальный запуск собранного дистрибутива для проверки.
* `npm run test` — запуск тестового набора Vitest.
* `npm run lint` — статический анализ кода на соответствие правилам ESLint.

## Тестирование

Для обеспечения стабильности интерфейсов и логики проект покрыт модульными и интеграционными тестами (Vitest + React Testing Library). Тестированию подвергаются:

* **Кастомные хуки** (проверка жизненного цикла и стейта).
* **Презентационные компоненты** (корректность рендеринга данных).
* **Интеграционные сценарии страниц** (взаимодействие каталога с контекстом фильтрации).
* **HOC-обертки** приватных роутов.

## Специфика деплоя на Vercel

Поскольку файл `.env.local` находится в `.gitignore` и не передается в удаленный репозиторий, при деплое на платформу Vercel необходимо:

1. В интерфейсе Vercel Dashboard перейти в **Settings** -> **Environment Variables**.
2. Задать все переменные, перечисленные в `.env.example`.
3. Запустить принудительную пересборку (**Redeploy**) из вкладки **Deployments**, чтобы сборщик Vite вшил новые переменные в клиентский бандл на этапе компиляции.
